> [!NOTE] Описание
> 1. Если в пути указано только доменное имя, то nginx сначала ищет файл index.html указанный в переменной index.
> 2. Если в пути указан файл, который реально существует, то он его откроет.
> 3. Потом ищет в директории, и если в ней так же есть файл index.html, то он его открывает.
> 4. А если ни какого пути не найдено, то nginx возвращает ошибку 404

***location /upload*** ^b39935


![[Pasted image 20260423103503.png]]


> [!NOTE] Описание
> Данная конструкция служит для безопасной отдачи файлов из каталога `upload`. А именно: напрямую при запросе картинки из каталога upload (по url адресу) не получится, т.к. nginx запрос к папке upload перенаправляет на скрипт php, который в свою очередь после выполнения запускает отдать файл из директивы internal_upload, а данная директива защищена от прямого доступа.

✦ Эта конфигурация описывает классическую схему работы Nginx с PHP-FPM для обработки загрузки файлов и их последующей защищенной отдачи (механизм X-Accel-Redirect).

  1. Блок location /upload/ (Обработка запроса)
  Этот блок перехватывает внешние запросы на адрес your-site.ru/upload/.

   * include /etc/nginx/fastcgi_params;: Подключает стандартный набор переменных (таких как QUERY_STRING, REQUEST_METHOD), которые Nginx передает в PHP.
   * fastcgi_param SCRIPT_FILENAME $document_root/upload.php\$args;: Это ключевая строка. Она говорит Nginx: "Какой бы файл внутри /upload/ ни запросил пользователь, на самом деле всегда запускай скрипт upload.php в корне сайта". 
       * Это часто используется для того, чтобы PHP-скрипт мог проверить права доступа пользователя перед тем, как разрешить загрузку или просмотр файла.
   * fastcgi_pass php-fpm;: Отправляет запрос на обработку в сервис PHP-FPM (через upstream, названный php-fpm).

2. Блок location /internal_upload/ (Служебный блок)
  Этот блок предназначен для эффективной отдачи файлов с диска силами самого Nginx.

   * internal;: Самая важная директива. Она делает этот блок "невидимым" для внешнего мира. Если вы введете в браузере your-site.ru/internal_upload/file.jpg, Nginx вернет ошибку 404. Доступ сюда можно получить только "изнутри"
     самого сервера (например, перенаправлением из PHP).
   * alias /srv/www/html/.../public/upload/;: Указывает реальный путь на диске, где лежат файлы.

✦ Как это работает вместе (Сценарий X-Accel-Redirect)

   1. Клиент просит файл: GET /upload/photo.jpg.
   2. Nginx (согласно первому блоку) передает этот запрос в PHP-скрипт upload.php.
   3. PHP делает проверки: "Авторизован ли пользователь? Оплатил ли он подписку?".
   4. Если всё хорошо, PHP не читает файл сам (это нагружает память), а отправляет специальный заголовок:
      X-Accel-Redirect: /internal_upload/photo.jpg.
   5. Nginx видит этот заголовок, понимает, что это внутренний запрос, идет во второй блок (/internal_upload/), берет файл напрямую с диска по адресу /srv/www/html/.../public/upload/photo.jpg и отдает его клиенту.

 ✦ **Итог: Вы получаете безопасность (проверка прав в PHP) и высокую производительность (отдача тяжелых файлов через Nginx, а не через PHP).**


> [!tip] Пример с сайта:
> Мы сможем получить доступ ко всем картинкам из каталога только если пройдем авторизацию, т.к. как я понимаю скрипт upload.php проверяет только это
