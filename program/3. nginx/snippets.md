#### deny-files.conf

```
# . fileslocation ~ /\.(?!well-known|vite) {  
  deny all;  
}  
  
# .git fileslocation ~ /\.git {  
  deny all;  
}
```

**Эти записи в конфиге Nginx — правила безопасности, предназначенные для того, чтобы скрыть от посторонних глаз служебные файлы и папки.**
___
1. Блок . files (Скрытые системные файлы)

   1 location ~ /\.(?!well-known|vite) {
   2   deny all;
   3 }
  Этот блок запрещает доступ ко всем файлам и папкам, которые начинаются с точки (скрытые файлы), кроме некоторых исключений.

   * location ~: Означает использование регулярного выражения (чувствительного к регистру).
   * /\.: Ищет путь, начинающийся с точки (например, .env, .htaccess, .ssh).
   * (?!well-known|vite): Это "отрицательное заглядывание вперед" (negative lookahead). Оно говорит: "Запрети всё, что начинается с точки, ЕСЛИ после точки НЕ идет well-known или vite".
       * .well-known: Необходима для работы SSL-сертификатов (Let's Encrypt проверяет владение доменом через эту папку).
       * .vite: Используется современным инструментом сборки фронтенда (Vite) для хранения временных файлов или конфигураций.
   * deny all: Закрывает доступ. Если кто-то попробует открыть example.com/.env, он получит ошибку 403 Forbidden.

  Зачем это нужно? Чтобы злоумышленники не могли скачать ваш файл .env с паролями от базы данных или другие системные конфиги.

  ---

  2. Блок .git files (Данные Git)

   1 location ~ /\.git {
   2   deny all;
   3 }
  Этот блок специально нацелен на папку .git.

   * /\.git: Ищет любые пути, содержащие .git (например, /project/.git/config).
   * deny all: Запрещает доступ.

  Зачем это нужно? Если папка .git доступна извне, хакер может скачать всю историю разработки вашего проекта вместе с исходным кодом, комментариями разработчиков и, иногда, забытыми в истории ключами доступа.

#### service-error.conf

✦ Этот блок конфигурации отвечает за отображение пользовательской страницы ошибки, когда ваш бэкенд (например, Python, Node.js или PHP) недоступен.

  Вот подробный разбор:

  1. Строка error_page 502 503 504 /502.html;
  Эта директива говорит Nginx: «Если произойдет одна из этих ошибок, не показывай стандартное белое окно, а перенаправь пользователя на внутренний адрес /502.html».

   * 502 (Bad Gateway): Бэкенд выключен или упал.
   * 503 (Service Unavailable): Бэкенд перегружен или находится на обслуживании.
   * 504 (Gateway Timeout): Бэкенд ответил слишком поздно (тайм-аут).

  2. Блок location = /502.html
  Здесь описывается, где именно лежит файл этой страницы и как с ним работать.

   * location = /502.html: Знак = означает точное соответствие. Это правило сработает только для этого конкретного пути.
   * root /etc/nginx/html;: Указывает папку на диске, где лежит файл. В данном случае Nginx будет искать файл по пути /etc/nginx/html/502.html.
   * internal;: Важный параметр безопасности. Он означает, что эта страница доступна только для внутренних перенаправлений самого Nginx. 
       * Если обычный пользователь введет в браузере ваш-сайт.com/502.html, он получит ошибку 404. 
       * Страница покажется только тогда, когда сам сервер решит выдать ошибку 502, 503 или 504.
  ---

  Зачем это нужно?

   1. Пользовательский опыт (UX): Вместо пугающей технической надписи "502 Bad Gateway" вы можете показать красивую страницу с логотипом вашей компании, извинениями и контактами техподдержки.
   2. Дизайн: Вы можете оформить страницу ошибки в стиле вашего основного сайта.
   3. Безопасность: Стандартные страницы Nginx часто выдают версию сервера (например, nginx/1.18.0). Своя страница скрывает эти подробности от потенциальных злоумышленников.

  Пример того, как это работает:
   4. Пользователь заходит на сайт.
   5. Ваш бэкенд-сервер (например, Node.js) упал.
   6. Nginx пытается передать запрос, получает отказ и генерирует ошибку 502.
   7. Nginx видит правило error_page и ищет файл /etc/nginx/html/502.html.
   8. Пользователь видит вашу красивую страницу "Мы скоро вернемся!".

#### ssl.conf

✦ Этот блок конфигурации отвечает за современную, безопасную и производительную настройку SSL/TLS. Она основана на рекомендациях Mozilla Intermediate, которые обеспечивают отличный баланс между высокой безопасностью и совместимостью со старыми браузерами.

  Вот подробный разбор каждой группы параметров:

  1. Оптимизация сессий (Производительность)
  Эти параметры ускоряют повторные подключения пользователей, избавляя их от необходимости выполнять полный (тяжелый) процесс "рукопожатия" (handshake).
   * ssl_session_timeout 1d;: Браузер может использовать данные старой сессии в течение 1 дня, не создавая новые ключи шифрования с нуля.
   * ssl_session_cache shared:SSL:10m;: Создает общую память (10 МБ, хватит примерно на 40 000 сессий) для хранения ID сессий. Это значительно снижает нагрузку на процессор при большом количестве посетителей.
   * ssl_session_tickets off;: Отключает билеты сессий. Это повышает безопасность (Perfect Forward Secrecy), так как не позволяет злоумышленнику расшифровать старый трафик, если он когда-то завладеет ключом билета.

  2. Обмен ключами (Diffie-Hellman)
   * ssl_dhparam /etc/pki/tls/dhparam.pem;: Указывает файл с параметрами Диффи-Хеллмана. Стандартные параметры в Nginx (1024 бит) считаются слабыми. Использование своего файла на 2048 или 4096 бит защищает от взлома ключей методом перебора.

  3. Протоколы и Шифры (Безопасность)
   * ssl_protocols TLSv1.2 TLSv1.3;: Запрещает старые и дырявые протоколы (SSLv3, TLS 1.0, 1.1). Оставляет только современные и безопасные.
   * ssl_ciphers ...;: Список разрешенных алгоритмов шифрования. Здесь подобраны самые надежные варианты (например, AES-GCM и ChaCha20), которые устойчивы к известным атакам.

  4. OCSP Stapling (Проверка сертификата)
  Обычно браузер сам обращается к издателю сертификата (CA), чтобы проверить, не отозван ли он. Это замедляет загрузку.
   * ssl_stapling on; (в коде закомментировано): Позволяет вашему серверу самому запрашивать статус сертификата у издателя и отдавать его браузеру вместе с сайтом ("пристегивать" — stapling). Это ускоряет загрузку страницы.
   * ssl_stapling_verify on;: Заставляет Nginx проверять ответы от издателя сертификата, чтобы убедиться в их подлинности.
   * resolver 172.16.65.69 valid=60s;: IP-адрес DNS-сервера, который Nginx будет использовать, чтобы найти сервер издателя сертификата для получения OCSP-статуса.

  Итог:
  Эта настройка делает ваш сайт:
   1. Защищенным от большинства известных атак на SSL (POODLE, BEAST и др.).
   2. Быстрым за счет кеширования сессий и OCSP Stapling.
   3. Современным (вы получите оценку A+ в тесте SSL Labs (https://www.ssllabs.com/ssltest/)).

✦ Процесс поиска сервера издателя (OCSP-респондера) происходит автоматически, но для этого Nginx проходит через несколько этапов. Вот как это работает «под капотом»:

  1. Информация внутри самого сертификата
  Когда вы покупаете или выпускаете SSL-сертификат (например, через Let's Encrypt или Sectigo), издатель записывает в него специальные метаданные. Одно из полей называется AIA (Authority Information Access).

  В этом поле указан прямой URL-адрес службы проверки, например:
  OCSP - URI: http://ocsp.int-x3.letsencrypt.org

  2. Как Nginx узнаёт адрес
  Когда вы включаете ssl_stapling on;, Nginx при запуске или перезагрузке читает ваш файл сертификата и «вытаскивает» оттуда этот URL. 

  3. Роль DNS Resolver
  Вот здесь вступает в дело ваша настройка resolver 172.16.65.69;:
   4. Nginx видит доменное имя из сертификата (например, ocsp.int-x3.letsencrypt.org).
   5. Чтобы отправить запрос этому серверу, Nginx нужно узнать его IP-адрес.
   6. Nginx отправляет DNS-запрос на адрес 172.16.65.69 (ваш внутренний или публичный DNS-сервер), спрашивая: «Какой IP у этого OCSP-сервера?».
   7. Получив IP, Nginx делает HTTP-запрос к издателю.

  8. Что такое Stapling («Пришпиливание»)
   9. Nginx получает от издателя короткий, подписанный цифровой подписью ответ: «С этим сертификатом всё ок, он не отозван. Годен до: [дата]».
   10. Nginx сохраняет (кеширует) этот ответ у себя.
   11. Когда на ваш сайт заходит пользователь, Nginx вместе с вашим сайтом сразу отдает ему и этот «чек» от издателя.
   12. Браузеру пользователя больше не нужно самому идти в интернет и проверять ваш сертификат — он верит «чеку», который приложил Nginx.

  Почему это важно?
   * Скорость: Пользователю не нужно ждать, пока его браузер свяжется с сервером издателя (это экономит 100-300 мс при первом соединении).
   * Приватность: Издатель сертификата не узнает IP-адреса ваших пользователей, потому что запросы к нему делает только ваш сервер, а не браузеры клиентов.
   * Надежность: Если DNS-резолвер не настроен или не работает, Nginx не сможет обновить статус сертификата, и в логах появится ошибка OCSP responder timed out.

  Зачем нужен параметр valid=60s?
  resolver 172.16.65.69 valid=60s; говорит Nginx: «Запоминай IP-адрес сервера издателя на 60 секунд. Если он изменится в DNS, через минуту переспроси заново». Это защищает от ситуаций, когда у издателя меняются серверы.

#### cache.conf

```
# assets, media
location ~* \.(?:css(\.map)?|js(\.map)?|jpe?g|png|gif|ico|cur|heic|webp|tiff?|mp3|m4a|aac|ogg|midi?|wav|mp4|mov|webm|mpe?g|avi|ogv|flv|wmv|geojson)$ {
    expires    7d;
    access_log on;
}

# svg, fonts
location ~* \.(?:svgz?|ttf|ttc|otf|eot|woff2?)$ {
    add_header Access-Control-Allow-Origin "*";
    expires    7d;
    access_log on;
}

```

✦ Эти два блока в конфигурации Nginx предназначены для оптимизации доставки статических файлов (картинок, видео, скриптов и шрифтов).

  Вот подробный разбор того, что они делают:

  ---

  Блок 1: Медиафайлы, скрипты и стили

   1 location ~* \.(?:css(\.map)?|js(\.map)?|jpe?g|png|gif|ico|cur|heic|webp|tiff?|mp3|m4a|aac|ogg|midi?|wav|mp4|mov|webm|mpe?g|avi|ogv|flv|wmv|geojson)$ {
   2     expires    7d;
   3     access_log on;
   4 }

   1. location ~*: Использовать регулярное выражение, игнорируя регистр (чтобы .JPG и .jpg обрабатывались одинаково).
   2. Список расширений:
       * css(\.map)? и js(\.map)?: Обрабатывает файлы стилей и скриптов, а также их Source Maps (.js.map). Это файлы, которые помогают разработчикам отлаживать минифицированный код в браузере.
       * Огромный список расширений для изображений (webp, png, jpg), аудио (mp3, wav) и видео (mp4, avi).
   3. expires 7d;: Устанавливает заголовок Cache-Control на 7 дней. 
       * Браузер пользователя скачает картинку один раз и в течение недели даже не будет спрашивать сервер о ней, а сразу возьмет из своего кеша. Это значительно ускоряет загрузку сайта при повторных посещениях.
   4. access_log on;: Включает запись запросов к этим файлам в лог-файл Nginx. (Иногда для статики лог выключают off, чтобы сэкономить место на диске, но здесь он явно включен).

  ---

  Блок 2: Шрифты и векторная графика (SVG)

   1 location ~* \.(?:svgz?|ttf|ttc|otf|eot|woff2?)$ {
   2     add_header Access-Control-Allow-Origin "*";
   3     expires    7d;
   4     access_log on;
   5 }

  Этот блок обрабатывает шрифты (woff2, ttf, otf) и векторные изображения (svg, svgz). У него есть одна критически важная настройка:

   1. add_header Access-Control-Allow-Origin "*"; (CORS):
       * Это разрешает загрузку этих файлов с других доменов. 
       * Почему это важно именно здесь? Браузеры по умолчанию блокируют загрузку шрифтов с другого домена из соображений безопасности. Если ваш сайт использует CDN или если шрифты подгружаются через JavaScript с другого поддомена, без этого заголовка
         шрифты просто не отобразятся («заквадратики»).
   2. expires 7d;: Также кеширует шрифты на неделю (шрифты меняются редко, поэтому их выгодно кешировать).

  ---

  Итог: зачем это нужно?

   1. Скорость: Вы "говорите" браузеру хранить тяжелые файлы у себя неделю.
   2. Экономия ресурсов: Сервер тратит меньше процессорного времени и трафика на отдачу одних и тех же картинок.
   3. Корректность отображения: Заголовок Access-Control-Allow-Origin гарантирует, что ваши иконки и шрифты не "сломаются" при загрузке.
   4. Отладка: Поддержка .map файлов позволяет комфортно смотреть ошибки в консоли разработчика, даже если сайт работает на сжатых (минифицированных) файлах.
